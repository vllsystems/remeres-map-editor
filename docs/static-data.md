# Static Data

This document describes the `staticdatahouse.dat` file exported by this Remere's Map Editor fork, what it is used for, which fields it contains, and the recommended workflow between map, server, and client.

## Overview

This fork exports a protobuf file named `staticdatahouse.dat`.

Main goals:

- provide static house data to the client without relying on `houses.xml`
- allow Cyclopedia house previews without generating PNG files
- keep a simple minimap-color fallback through `colors`
- support modern preview rendering through `tiles`

The file is static because it does not depend on server runtime state. It represents data that comes from the map itself:

- house identity
- house name
- city
- exit position
- square meters
- beds
- base rent
- guildhall / shop flags
- visual preview

## What it does not replace

`staticdatahouse.dat` does not replace dynamic information sent by the server.

Examples of data that must still come from the server:

- current house status
- highest bid
- whether the player can bid
- reason why bidding is blocked
- current ownership
- auction timers and dates

In short:

- `staticdatahouse.dat` = static metadata and preview data
- server packets = current runtime state

## Relation to `houses.xml`

`houses.xml` is still useful for the server, but it does not contain enough information for a modern visual preview in the client.

Examples of what `houses.xml` does not solve by itself:

- which tiles belong in the preview
- which items must be rendered
- `clientId` for each item
- `patternX`, `patternY`, `patternZ`
- a mask telling which tiles belong to the actual house and which belong only to surrounding context

Because of that, the preview should come from the map export generated by RME, not from `houses.xml`.

## Where the file is generated

Current export:

- `staticdatahouse.dat`

The filename is defined in:

- `source/iomap_otbm.cpp`

In the current implementation, this file contains house data only.

## Protobuf structure

Source file:

- `source/protobuf/staticdata.proto`

Top-level structure:

```proto
message Staticdata {
  repeated House houses = 3;
}
```

Each `House` exports:

- `id`
- `name`
- `description`
- `rent`
- `beds`
- `position`
- `SQMs`
- `isGuildHall`
- `city`
- `isShop`
- `preview`

## House fields

### `id`

Map house id. It should be unique and stable.

### `name`

House name from the map.

### `description`

Currently exported as empty.

### `rent`

Base rent configured on the house.

### `beds`

Number of beds assigned to the house.

### `position`

Exported from the house exit position:

- `x`
- `y`
- `z`

### `SQMs`

Exported from `house->size()`.

### `isGuildHall`

Direct guildhall flag from the house.

### `city`

Town name resolved from the house `townid`.

### `isShop`

Currently inferred from the house name.

Current rule:

- names containing `(Shop)` or `(shop)` are exported as shop houses

## Preview structure

The preview is exported in `HousePreview`.

Fields:

- `width`
- `height`
- `floor`
- `colors`
- `tiles`

### `width` and `height`

These define the exported preview area dimensions.

Important:

- they do not represent only the house interior
- they also include nearby context

Current implementation exports a margin of `2` tiles around the selected house area on the chosen floor.

### `floor`

Floor used for the preview.

Current selection rule:

1. if the house exit floor exists among the house tiles, that floor is used
2. otherwise, the floor with the largest number of house tiles is used

### `colors`

Linear matrix of minimap colors, one byte per tile.

Recommended use:

- legacy fallback
- quick debugging
- compatibility with simple preview renderers

This is not the ideal format for a modern preview.

### `tiles`

This is the primary format for modern preview rendering.

Each tile exports:

- `x`
- `y`
- `color`
- `items`
- `isHouseTile`

### `isHouseTile`

Indicates whether the tile belongs to the actual house area.

Recommended client behavior:

- render the entire preview area
- darken or tint tiles outside the actual house

This enables a CIP-like result:

- surrounding context remains visible
- the actual house area remains visually highlighted

## Preview items

Each preview tile may export up to `5` items.

Each item exports:

- `clientId`
- `xPattern`
- `yPattern`
- `zPattern`

This is enough for OTC to render the preview using real `ThingType` data instead of PNG files.

### Exported render order

The exporter assembles tile items in an order intended for visual preview rendering:

1. ground
2. bottom items
3. common items
4. top items

This allows the client to reproduce a stable visual result without loading the full live map.

### Special cases handled by the exporter

The current exporter adjusts patterns for common item types:

- stackables
- hangables
- splash
- fluid containers

So the file does not store only `clientId`; it also stores the sprite frame needed to make the preview look correct.

## How the preview is built

Simplified flow:

1. select the preview floor
2. calculate the minimum bounding rectangle of the house on that floor
3. expand the rectangle with a context margin
4. iterate through all tiles in that area
5. save:
   - minimap color
   - renderable items
   - `isHouseTile` mask

If no useful preview data is found, no preview is exported for that house.

## Recommended client usage

Recommended OTC flow:

1. try to load `staticdatahouse.dat`
2. use `preview.tiles` as the primary data source
3. use `preview.colors` only as a legacy fallback
4. combine static preview data with dynamic house data received from the server

Recommended client location:

- `data/things/<version>/assets/staticdatahouse.dat`

If the client supports additional fallbacks, that is fine, but the path above should be the primary one.

## Best practices

### 1. Treat `staticdatahouse.dat` as a map build artifact

Whenever house data changes in the map, re-export the file.

Examples:

- house area changed
- town changed
- exit changed
- bed count changed
- house name changed
- structural decoration changed in a way that affects the preview

### 2. Do not build previews from PNG files

PNG is a derived asset, harder to maintain, and a poor fit for custom servers.

Prefer:

- `tiles` for modern rendering
- `colors` only as fallback

### 3. Do not rely on `houses.xml` for visual preview

`houses.xml` is suitable for textual and server-side data, but not for accurate visual preview rendering.

### 4. Keep `clientId` compatible with the target client

The preview depends on `clientId` and pattern values.

If the map is exported with one asset set and opened by a client using different assets, the preview may render incorrectly.

### 5. Re-export after asset changes

If `dat` or sprite changes affect `clientId`, re-export `staticdatahouse.dat`.

### 6. Use `isHouseTile` to highlight the actual house

The best visual result is not to crop context away.

Instead:

- keep surrounding tiles visible
- apply tint or overlay outside the house area

## Current limitations

In the current implementation:

- the file exports houses only
- `description` is empty
- `isShop` is inferred from the house name
- preview uses a single floor per house
- each tile exports at most `5` items

These constraints are intentional to keep the file compact and the client renderer simple.

## When to change the protobuf

Edit `source/protobuf/staticdata.proto` only when the client genuinely needs additional static data.

Before adding a new field, validate:

1. does this data come from the map itself?
2. does it need to exist offline on the client?
3. should it really be static, instead of coming from the server at runtime?

Practical rule:

- structural or visual map data -> good candidate for `staticdatahouse.dat`
- gameplay state -> should come from the server

## Recommended integration flow

### RME

- edit the map
- configure houses correctly
- export the map
- generate `staticdatahouse.dat`

### Server

- use `houses.xml` and map data for house logic
- send only dynamic house state through the protocol

### Client

- load `staticdatahouse.dat`
- render HD preview through `tiles`
- use `colors` only as fallback
- combine static data with dynamic server data for UI and actions

## Summary

`staticdatahouse.dat` exists to decouple static house metadata and visual preview data from dynamic server state.

The core idea is:

- the map defines structure and appearance
- the server defines current state
- the client combines both

For custom servers, this is the most flexible and maintainable approach.
